## 背景：为什么有这份笔记

被动采集 MVP 已跑通（浏览 B 站视频页即自动入库）。下一阶段的核心诉求（用户原话）：

> "在服务端控制需求，能让 AI 去搜索需要的内容，控制速度，然后搜集到数据。"

即：**把 collector-server 从"被动接收器"升级成"采集大脑/控制中心"**——
服务端表达需求 → 自己拉视频列表 → 控制采集速度 → 驱动采集终端逐个采集入库 → 进度可见。

这正是旧 spec §4 预留的"任务调度层" + §9 推迟的"批量/AI 命令采集"。

---

## 已确认的约束 / 起点

1. **服务端做控制中心** —— 调度逻辑放在 collector-server，不在扩展。
2. **要能控制速度** —— 限速是核心需求（防风控、防开一堆 tab）。
3. **要支持"AI 搜索需求"** —— 但 AI 的具体角色/接入方式待定（见下）。
4. **不动现有被动采集链路** —— 浏览即入库已可用，主动采集是叠加能力。
5. **现有 WS 命令通道已预留** —— `navigate` / `operate` / `fetch-subtitle` 三个 Command 在 spec §6.2/§7.3 预留，background.js 已实现 navigate/operate 骨架，fetch-subtitle 是占位。

## 核心架构岔路（待拍板）

"AI 在服务端搜索需求并控制采集"有三种实现路径，影响整个 spec 走向：

### A. 结构化任务 API（服务端调度器 + AI 作外部调用方）—— **倾向推荐**

- 服务端新增 `/api/tasks`（任务类型：keyword / upper / list / 时间段）+ 内置**调度器**。
- 调度器：拉视频列表 → 限速下发 navigate 给采集终端 → 收 ingest → 更新进度 → 失败重试/跳过。
- AI/你 = **外部调用方**，发一个任务就放手；可用 MCP 工具 / CLI / 网页按钮任一驱动同一个 task API。
- **优点**：决策与执行解耦；"控制速度"本质是调度器，A 和 B 都需要；AI 可后接（包一层 MCP 工具即可），调度器一行不改。
- **缺点**：需要先设计调度器/任务表/限速策略，有一定工程量。

### B. AI Agent 对话循环（服务端内置 LLM 决策）

- 服务端跑对话循环：AI 收到自然语言需求 → 自己调 search/collect 工具 → 自己决定采哪个 → 结果返回。
- 需在服务端暴露 MCP 工具 + 调 LLM + 管理对话状态 + 处理幻觉/成本/超时。
- **优点**：最"智能"，自然语言直达。
- **缺点**：复杂度高；LLM 成本/幻觉/超时与采集逻辑耦合；决策不确定性大。

### C. 先手走通再定（脚本实验驱动）

- 写脚本手走"拉列表→逐个 navigate→观察"一遍，记录痛点（速度/失败/风控），再据此定架构。
- **优点**：需求不明时最稳。
- **缺点**：不可复用，只是探路。

**初步判断：A 是 B 的前置依赖。** 无论 AI 在哪决策，"拿到列表→限速→逐个采集→记录进度"的调度引擎都必须存在。先做 A，将来想上 AI 对话只需在 A 之上加 MCP/Agent 层，调度器不动。

## 另一个待拍板点：采集终端是谁

批量采集时谁来提供登录态、执行开页采集：

| 选项 | 说明 | 代价 |
|---|---|---|
| **复用现有扩展** | 调度器下发 navigate → 扩展在你当前 Chrome 开页采集 | 用现有登录态/Cookie，最简单；批量时会在当前浏览器开一堆 tab（可控：采完自动关） |
| **服务端独立 headless** | 服务端跑 puppeteer 登录态 profile，不碰日常浏览器 | 更"后台"、不打扰；但要管 profile/登录态/Cookie（opencli 走这条） |
| **服务端直调 API** | 不靠浏览器，服务端直接调 B 站 API 拉字幕（opencli bilibili/subtitle.js 路线） | 绕过浏览器；但要用 Cookie + 处理 Wbi 签名坑 |

## 还没展开的设计点（恢复 brainstorming 时逐个过）

- 调度器内部模型：任务表 schema、任务状态机、进度计算、并发/限速参数（每秒几个、并发几个 tab）。
- 视频列表来源：B 站 search API（关键词）/ UP主 `arc/search`（时间段过滤）/ 手贴列表。Wbi 签名坑（被动采集天然绕过，主动拉列表需自行签名）。
- 限速与风控：触发风控后怎么办（暂停 N 分钟？换终端？）；进度持久化（服务重启续跑？）。
- 进度反馈：collector-web 加任务列表/详情页？还是只 API？popup 要不要反映服务端任务？
- fetch-subtitle Command 补实现（spec §6.2 占位），服务端能精确驱动"某 bvid 的字幕抓取"。
- popup 现有空壳修复（"当前视频/上报"两行 + 手动补采反馈）——这块偏 UX 收尾，可与主动采集解耦单独做。
